## Introduction

**LongLive 1.0**: Real-time Interactive Long Video Generation. [You can find it here](https://github.com/NVlabs/LongLive/tree/v1.0) in our V1.0 branch.

**LongLive 2.0**: an NVFP4 Parallel Infrastructure for Long Video Generation
- For training, it supports
  - [x] Balanced sequence parallel for T2V/I2V AR training (teacher-forcing).
  - [x] T2V/I2V AR training on multi-shot (or single-shot) videos.
  - [x] NVFP4 (or BF16) for both AR training and few-step distillation.
- For inference, it supports
  - [x] NVFP4 inference (W4A4) and NVFP4 KV Cache.
  - [x] Multi-shot attention sink.
  - [x] Sequence parallel inference.
  - [x] Async decoding.

#### BF16

```python
import torch
from omegaconf import OmegaConf

from pipeline import CausalDiffusionInferencePipeline
from utils.config import normalize_config
from utils.inference_utils import (
    load_generator_checkpoint,
    place_vae_for_streaming,
    prepare_single_prompt_inputs,
    save_video,
)

prompt = "A compact silver robot walks through a clean robotics lab."
merged_checkpoint_path = "LongLive-2.0-5B/model_bf16.pt"

config = normalize_config(OmegaConf.load("configs/inference.yaml"))
device = torch.device("cuda")

torch.set_grad_enabled(False)
pipe = CausalDiffusionInferencePipeline(config, device=device)
load_generator_checkpoint(pipe.generator, merged_checkpoint_path)
pipe = pipe.to(device=device, dtype=torch.bfloat16)
place_vae_for_streaming(pipe, config)  # honor streaming_vae + vae_device when set
pipe.generator.model.eval().requires_grad_(False)

noise, prompts = prepare_single_prompt_inputs(config, prompt, device)
video = pipe.inference(noise=noise, text_prompts=prompts)
save_video(video[0], "videos/quickstart/sample.mp4", fps=24)
```

`place_vae_for_streaming` is a no-op unless `inference.streaming_vae` is true and `inference.vae_device` is set, so toggling streaming-pipeline decode in your yaml is enough — the script does not need to change.

#### NVFP4

Point `checkpoints.generator_ckpt` in `configs/nvfp4/inference_nvfp4.yaml` at the downloaded checkpoint and set `model_quant_use_transformer_engine` according to the backend you are using:

- TransformerEngine checkpoint (`model_te.pt`): `model_quant_use_transformer_engine: true`
- FourOverSix checkpoint (`model_4o6.pt`): `model_quant_use_transformer_engine: false`

`setup_nvfp4_pipeline` handles checkpoint loading, NVFP4 module wrapping, weight materialization, dtype/device placement, and the streaming-pipeline VAE relocation for both backends — the bf16 `pipe.to(...)` shortcut is unsafe here because it would cast the quantized buffers.

```python
import torch
from omegaconf import OmegaConf

from pipeline import CausalDiffusionInferencePipeline
from utils.config import normalize_config
from utils.inference_utils import prepare_single_prompt_inputs, save_video, setup_nvfp4_pipeline

prompt = "A compact silver robot walks through a clean robotics lab."

config = normalize_config(OmegaConf.load("configs/nvfp4/inference_nvfp4.yaml"))
device = torch.device("cuda")

torch.set_grad_enabled(False)
pipe = CausalDiffusionInferencePipeline(config, device=device)
setup_nvfp4_pipeline(pipe, config, device)
pipe.generator.model.eval().requires_grad_(False)

noise, prompts = prepare_single_prompt_inputs(config, prompt, device)
video = pipe.inference(noise=noise, text_prompts=prompts)
save_video(video[0], "videos/quickstart/sample_nvfp4.mp4", fps=24)
```

## Training Modes

LongLive2.0 supports both T2V and I2V training. Each modality follows the same two-stage recipe: AR teacher-forcing training first, then DMD distillation from the AR checkpoint.

### T2V Training

```bash
torchrun --standalone --nnodes=1 --nproc_per_node=8 train.py \
  --config_path configs/train_ar.yaml \
  --logdir logs/train_ar \
  --wandb-save-dir wandb \
  --disable-wandb

torchrun --standalone --nnodes=1 --nproc_per_node=8 train.py \
  --config_path configs/train_dmd.yaml \
  --logdir logs/train_dmd \
  --wandb-save-dir wandb \
  --disable-wandb
```

### I2V Training

```bash
torchrun --standalone --nnodes=1 --nproc_per_node=8 train.py \
  --config_path configs/train_i2v_ar.yaml \
  --logdir logs/train_i2v_ar \
  --wandb-save-dir wandb \
  --disable-wandb

torchrun --standalone --nnodes=1 --nproc_per_node=8 train.py \
  --config_path configs/train_i2v_dmd.yaml \
  --logdir logs/train_i2v_dmd \
  --wandb-save-dir wandb \
  --disable-wandb
```

For I2V configs, set `algorithm.i2v: true` and `algorithm.independent_first_frame: true`. `data.image_or_video_shape[1]` is the full latent sequence length, for example `96`, not `96 + 1`: the clean image latent replaces the first latent during denoising and that first latent is masked out of the training loss. For I2V DMD, set `checkpoints.generator_ckpt` to the I2V AR checkpoint used to initialize the student.

### Camera (PRoPE) Bidirectional SFT

This stage adds **Bidirectional + camera (PRoPE) SFT** on top of the
Wan2.2-TI2V-5B backbone. It is the LongLive port of the corresponding minWM
recipe (`training_wan.md` §1.1) and reuses the original
`utils/scheduler.FlowMatchScheduler`. PRoPE is implemented as **zero-init
residual attention blocks** (see `wan_5b/modules/prope.py`), so before any
gradient step the camera-conditioned model behaves identically to the
pretrained Wan2.2-TI2V-5B — making this a safe, additive fine-tune.

Key files:

| Path | Purpose |
| --- | --- |
| `wan_5b/modules/prope.py` | PRoPE math + `add_prope_parameters()` |
| `wan_5b/modules/model.py` | `WanSelfAttention.forward(... prope_meta)` residual path |
| `utils/wan_5b_camera_wrapper.py` | `CameraWanDiffusionWrapper` (PRoPE-aware) |
| `utils/camera_dataset.py` | LMDB dataset returning `(latent, viewmats, Ks, prompt)` |
| `model/camera_bidirectional_diffusion.py` | Algorithm: bidirectional flow-matching loss |
| `trainer/camera_bidirectional_diffusion.py` | Self-contained FSDP trainer |
| `configs/train_bidir_camera.yaml` | Training config |
| `configs/infer_bidir_camera.yaml` | Inference config |

#### (1) Pre-download the Wan2.2-TI2V-5B base model

If you have already followed the LongLive 2.0 quick-start, skip this step.

```bash
HF_XET_HIGH_PERFORMANCE=1 hf download Wan-AI/Wan2.2-TI2V-5B \
    --local-dir wan_models/Wan2.2-TI2V-5B
```

#### (2) Prepare the dataset

Author a JSON manifest at `./dataset/LongLive/CameraSFT/raw/clips.json` of
the form:

```json
[
  {
    "video_path": "/abs/path/to/clip_0.mp4",
    "caption":    "A first-person walk through a sunlit forest...",
    "pose_str":   "w-30, left-10, w-30"
  },
  {
    "video_path": "/abs/path/to/clip_1.mp4",
    "caption":    "A drone-like flight forward over the sea...",
    "pose_str":   "w-76"
  }
]
```

`pose_str` follows the WorldPlayGen grammar: `w/s/a/d` for translate
forward / back / left / right, `up/down` for pitch, `left/right` for yaw,
each with a `-N` frame count, comma-separated. Defaults are
`forward_speed=0.08`, `yaw_speed=pitch_speed=3°` per frame.

Then build the LMDB (default 8-GPU; produces `./dataset/LongLive/CameraSFT/data/`):

```bash
INPUT_JSON=./dataset/LongLive/CameraSFT/raw/clips.json \
OUTPUT_DIR=./dataset/LongLive/CameraSFT \
TARGET_H=704 TARGET_W=1280 MAX_FRAMES=77 NPROC=8 \
    bash scripts/data_preprocessing/run_build_camera_lmdb_5b.sh
```

The script encodes 77 raw frames at `1280×704` with the Wan2.2 VAE
(4× temporal, 16× spatial, 48 channels) into a `(F_lat=20, 48, 44, 80)`
fp16 latent and stores per-record `latents_*`, `prompts_*`, `intrinsics_*`
(normalized `[fx/W, fy/H, cx/W, cy/H]`), `poses_*` (`[tx,ty,tz,qx,qy,qz,qw]`,
F_lat × 7).

#### (3) Train (bidirectional + PRoPE)

```bash
bash scripts/training/run_stage0_bidirectional_camera.sh
# or:
torchrun --standalone --nnodes=1 --nproc_per_node=8 train.py \
    --config_path configs/train_bidir_camera.yaml \
    --logdir logs/train_bidir_camera \
    --wandb-save-dir wandb \
    --disable-wandb
```

Default checkpoints land at
`./logs/train_bidir_camera/checkpoint_model_XXXXXX/model.pt`. Pick the best
step (typically by held-out flow-loss) and promote it to a stable path:

```bash
BEST_STEP=005000
mkdir -p ./checkpoints/LongLive-2.0-5B-Camera/
cp ./logs/train_bidir_camera/checkpoint_model_${BEST_STEP}/model.pt \
   ./checkpoints/LongLive-2.0-5B-Camera/model.pt
```

#### (4) Inference

Author one prompt per line in `prompts/prompts.txt` and one matching
trajectory string per line in `prompts/trajectories.txt` (samples are
provided). Then:

```bash
CKPT=./checkpoints/LongLive-2.0-5B-Camera/model.pt \
OUT=videos/camera_bidir \
    bash scripts/inference/run_infer_bidirectional_camera.sh

# or directly:
python scripts/inference/inference_bidir_camera.py \
    --config_path  configs/infer_bidir_camera.yaml \
    --generator_ckpt ./checkpoints/LongLive-2.0-5B-Camera/model.pt \
    --output_dir   videos/camera_bidir
```

Outputs are `.mp4` clips at 24 fps (`F_lat=20` latent frames decode to
77 raw RGB frames at `1280×704`), one per `(prompt, trajectory)` pair, with
a sibling `.txt` recording the prompt and trajectory used.

#### Notes on the implementation

- PRoPE is added as a **zero-init residual** path (`prope_o = nn.Linear(dim, dim)`
  with weight & bias zero), so at step 0 the model is bit-exact equal to the
  pretrained Wan2.2-TI2V-5B regardless of `viewmats`. Setting
  `algorithm.use_camera = false` (or omitting `viewmats`) bypasses the path
  entirely.
- All frames in a batch share a single timestep (`uniform_timestep = True`).
- The new trainer (`trainer/camera_bidirectional_diffusion.py`) only relies
  on LongLive's own `utils/distributed.fsdp_wrap`, `EMA_FSDP`, and
  `fsdp_state_dict`, so it composes with the rest of the repo with no
  changes to existing AR / DMD trainers.
