## Introduction

**LongLive 1.0**: Real-time Interactive Long Video Generation. [You can find it here](https://github.com/NVlabs/LongLive/tree/v1.0) in our V1.0 branch.

**LongLive 2.0**: an NVFP4 Parallel Infrastructure for Long Video Generation
- For training, it supports
  - [x] Balanced sequence parallel for T2V/I2V AR training (teacher-forcing).
  - [x] T2V/I2V AR training on multi-shot (or single-shot) videos.
  - [x] NVFP4 (or BF16) for both AR training and few-step distillation.
- For inference, it supports
  - [x] NVFP4 inference (W4A4) and NVFP4 KV Cache.
  - [x] Multi-shot attention sink.
  - [x] Sequence parallel inference.
  - [x] Async decoding.

##### DreamX Camera (E-PRoPE) I2V  + Echo-Infinity Memory + Relative RoPE -> teacher forcing AR

在 `configs/train_dreamx_camera_i2v_ar.yaml`（DreamX Camera AR 基线）基础上，叠加
Echo-Infinity 的「memory KV + Relative RoPE」。这样训出来的 checkpoint 同时具备：
camera 轨迹控制（E-PRoPE `cam_self_attn`）+ 无限长记忆（QueryMemoryEncoder）+
相对 RoPE。核心差异只有两块，且与 camera 分支完全正交、并联：

1. **QueryMemoryEncoder**：新增一组可学习 M-token query（默认 `Q_frames=8`，
   `M_tokens_per_frame=32`，共 256 token），在每个 chunk 结束时从被 evict 的
   KV 页里蒸馏出 memory KV，塞回 content self-attention 里，让被淘汰帧的信息仍能
   参与后续注意力。**不影响** `cam_self_attn`（camera 分支独立计算、独立 KV，
   不进 memory）。
2. **Relative RoPE**：把 `sink | memory | local | Q` 四段全部倒推到
   `[0, pmax-1]` 相对窗口（`Q` 钉在 `pmax-1`），避免绝对位置随视频长度线性
   漂移导致的注意力错位；当前 4-GPU 配置用 `pmax=32` 覆盖
   sink(4) + local(8) + memory(8) + block(4) 的最大展开（24 < 32）。

核心架构：每层 block 中两个**并行但不同语义**的分支同时存在——

```
block forward (_block_infmem_forward)
  │
  ├─ modulated_x = AdaLN(x, e_mod[0], e_mod[1])
  │
  ├─ 内容 self-attention (_self_attn_infmem_forward)
  │    ├─ sink KV       [固定低位置 0..sink-1]
  │    ├─ memory KV     [sink 后..mem_end]  ← QueryMemoryEncoder.get_kv()
  │    ├─ local KV      [local_start..local_end]
  │    ├─ current Q     [q_start..q_last]
  │    └─ Relative RoPE: 所有位置限制在 [0, pmax-1]
  │    → x += e_mod[2] * y_content
  │
  ├─ camera self-attention (cam_self_attn)  [仅当 prope_meta 存在]
  │    ├─ viewmats → prope_qkv → E-PRoPE
  │    ├─ Ks (intrinsics)
  │    └─ 使用相同 modulated_x，相同 e_mod[2] gate
  │    → x += e_mod[2] * y_cam
  │
  └─ cross-attention + FFN
```

初始化顺序：

1. 先调用 `DreamXCameraWanDiffusionWrapper.__init__` → 创建 causal Wan model + 每层 `cam_self_attn`
2. 加载 DreamX camera checkpoint（恢复 backbone + `cam_self_attn`）
3. 调用 `_attach_infmem_to_wrapper` → monkey-patch block/self_attn/model._forward_inference
4. 创建 `QueryMemoryEncoder` 并通过 `object.__setattr__` 挂载（绕开 FSDP flatten-params）

关键文件：

| 文件 | 作用 |
|---|---|
| `utils/dreamx_infmem_wrapper.py` | `DreamXInfMemWanDiffusionWrapper`：继承 `DreamXCameraWanDiffusionWrapper`（不继承 `InfMemWanDiffusionWrapper`，避免菱形继承），在 cam_self_attn 创建后挂载 InfMem patch + QueryMemoryEncoder |
| `utils/infinity_memory_wrapper.py` | 抽取共享 helper `_attach_infmem_to_wrapper`，供 `InfMemWanDiffusionWrapper` 和 `DreamXInfMemWanDiffusionWrapper` 共同调用；`InfMemWanDiffusionWrapper` 本身保持不变（纯 I2V AR，无 camera） |
| `wan_5b/modules/infinity_memory.py` | `_model_forward_inference_infmem` 增加 `viewmats`/`Ks` 参数和 `prope_meta` 构建（逻辑与 `CausalWanModel._forward_inference` 一致）；`_block_infmem_forward` 增加 `prope_meta` 参数，在跑完 memory-aware content self-attn 后，若 `block.cam_self_attn` 存在且 `prope_meta` 非空，照抄原 camera 残差分支（同一个 `modulated_x`、同一个 AdaLN gate `e_mod[2]`） |
| `model/query_memory.py` | 从 Echo-Infinity 迁移的 `QueryMemoryEncoder`（`reset / update / get_kv / detach_state`），`tokens_per_frame` 默认 880 对齐 5B 的 1280×704 latent 网格 |
| `model/diffusion.py` | `_generator_loss_infmem_streaming` 增加 `viewmats`/`Ks` 参数和相机切片、有效 `x0_pred` 返回；`_initialize_models` 通过 `wrapper_cls` 做 dotted-path dispatch |
| `pipeline/causal_diffusion_inference.py` | CFG 负分支和 noisy forward 传 `update_memory=False`，防止负分支污染 memory；`_camera_kwargs` 按 AR chunk 切片 `viewmats`/`Ks` |
| `trainer/diffusion.py` | `use_camera_lmdb` 探测到 camera 数据集时走 `_train_one_step_camera`，把 `viewmats`/`Ks` 传进 `generator_loss`；检测到 encoder 存在时开独立 param group（LR = `base_lr × encoder_lr_multiplier`） |
| `configs/train_dreamx_camera_i2v_ar_infmem.yaml` | 训练配置（基于 `train_dreamx_camera_i2v_ar.yaml`，保留全部 camera 设置） |
| `configs/infer_dreamx_camera_i2v_ar_infmem.yaml` | 推理配置（基于 `infer_dreamx_camera_i2v_ar.yaml`） |
| `tests/test_dreamx_infmem_combined.py` | 20 项 smoke test |

训练时每个 streaming chunk 的执行顺序：

1. 切片 camera：`chunk_viewmats = viewmats[:, start:end]`，`chunk_Ks = Ks[:, start:end]`
2. **noisy prediction forward**（`update_memory=False`，不推进 cache/memory）
3. **clean cache update forward**（`update_memory=True`，推进 KV cache + memory）
4. BPTT 截断（`maybe_detach_infmem`）

Checkpoint 初始化规则：

- **冷启动**：从 Bidirectional DreamX Camera I2V checkpoint 加载 backbone + `cam_self_attn`；`QueryMemoryEncoder` 新初始化；optimizer 新初始化
- **Resume**：恢复 generator（含 cam_self_attn）+ `QueryMemoryEncoder` + optimizer + EMA + step

向后兼容：

- 原 `train_i2v_ar.yaml` / `train_i2v_ar_infmem.yaml` / `train_dreamx_camera_i2v_ar.yaml` 行为不变
- `memory_kwargs=None` 时，combined wrapper 退化为 DreamX Camera AR（InfMem patch 在 `query_memory_encoder is None` 时直接调 `_original_forward`）
- 不传 `viewmats/Ks` 时，camera branch 被跳过，退化为普通 InfMem AR
- 已有 checkpoint tensor key 名（`blocks.{i}.cam_self_attn.*`）不变

建议先跑一次 `enable_relative_rope=false + memory_kwargs=null` 的 sanity run
比对 loss，确认 InfMem monkey-patch 无副作用，再打开 memory；camera 部分复用

```python
import torch
from omegaconf import OmegaConf

from pipeline import CausalDiffusionInferencePipeline
from utils.config import normalize_config
from utils.inference_utils import (
    load_generator_checkpoint,
    place_vae_for_streaming,
    prepare_single_prompt_inputs,
    save_video,
)

prompt = "A compact silver robot walks through a clean robotics lab."
merged_checkpoint_path = "LongLive-2.0-5B/model_bf16.pt"

config = normalize_config(OmegaConf.load("configs/inference.yaml"))
device = torch.device("cuda")

torch.set_grad_enabled(False)
pipe = CausalDiffusionInferencePipeline(config, device=device)
load_generator_checkpoint(pipe.generator, merged_checkpoint_path)
pipe = pipe.to(device=device, dtype=torch.bfloat16)
place_vae_for_streaming(pipe, config)  # honor streaming_vae + vae_device when set
pipe.generator.model.eval().requires_grad_(False)

noise, prompts = prepare_single_prompt_inputs(config, prompt, device)
video = pipe.inference(noise=noise, text_prompts=prompts)
save_video(video[0], "videos/quickstart/sample.mp4", fps=24)
```

`place_vae_for_streaming` is a no-op unless `inference.streaming_vae` is true and `inference.vae_device` is set, so toggling streaming-pipeline decode in your yaml is enough — the script does not need to change.

#### NVFP4

Point `checkpoints.generator_ckpt` in `configs/nvfp4/inference_nvfp4.yaml` at the downloaded checkpoint and set `model_quant_use_transformer_engine` according to the backend you are using:

- TransformerEngine checkpoint (`model_te.pt`): `model_quant_use_transformer_engine: true`

### Camera (PRoPE) Bidirectional SFT

This stage adds **Bidirectional + camera (PRoPE) SFT** on top of the
Wan2.2-TI2V-5B backbone. It is the LongLive port of the corresponding minWM
recipe (`training_wan.md` §1.1) and reuses the original
`utils/scheduler.FlowMatchScheduler`. PRoPE is implemented as **zero-init
residual attention blocks** (see `wan_5b/modules/prope.py`), so before any
gradient step the camera-conditioned model behaves identically to the
pretrained Wan2.2-TI2V-5B — making this a safe, additive fine-tune.

| `utils/wan_5b_camera_wrapper.py` | `CameraWanDiffusionWrapper` (PRoPE-aware) |

##### (2a) Use the minWM toy dataset

The minWM toy dataset is already downloaded at
`data/minWM-data/` with this layout:

```
data/minWM-data/
├── preencode_input.json          # list of {"caption", "pose_str"} (no video_path!)
└── videos/
    ├── 000000_right8a11/gen.mp4  # <index>_<compact-pose>/gen.mp4
    ├── 000001_left8w4a7/gen.mp4
    └── ...                       # each gen.mp4 is 77 frames @ 832×480
```

Two things make this dataset not directly consumable by the LMDB builder:

1. `preencode_input.json` has **only** `caption` and `pose_str` — there is
   **no** `video_path`. The matching clip lives at
   `videos/<index>_*/gen.mp4`, where `<index>` is the position of the entry
   in the JSON list.
2. The JSON lists far more captions (~20k) than the videos actually shipped
   in the toy bundle (~1k), so we only keep entries that have a video on disk.

The builder expects one JSON record per video of the form
`{"video_path", "caption", "pose_str"}`. Generate that manifest by joining
each `videos/<index>_*/gen.mp4` back to its caption/pose via the index:

```bash
python - <<'PY'
import json, os, glob
ROOT = "data/minWM-data"
meta = json.load(open(f"{ROOT}/preencode_input.json"))
clips = []
for mp4 in sorted(glob.glob(f"{ROOT}/videos/*/gen.mp4")):
    idx = int(os.path.basename(os.path.dirname(mp4)).split("_", 1)[0])
    if idx >= len(meta):
        continue
    clips.append({
        "video_path": os.path.abspath(mp4),
        "caption":    meta[idx]["caption"],
        "pose_str":   meta[idx]["pose_str"],
    })
json.dump(clips, open(f"{ROOT}/clips.json", "w"), ensure_ascii=False, indent=2)
print(f"wrote {len(clips)} clips -> {ROOT}/clips.json")
PY
```

This writes `data/minWM-data/clips.json` (≈1.2k entries on the current toy
bundle). `pose_str` follows the WorldPlayGen grammar: `w/s/a/d` for translate
forward / back / left / right, `up/down` for pitch, `left/right` for yaw,
each with a `-N` frame count, comma-separated. Defaults are
`forward_speed=0.08`, `yaw_speed=pitch_speed=3°` per frame.

> To use your **own** videos, just author a `clips.json` directly in the same
> `{"video_path", "caption", "pose_str"}` form — `video_path` may be absolute,
> relative to the repo root, relative to the JSON file, or relative to
> `VIDEO_DIR` below — and skip the snippet above.

The script loads `MAX_FRAMES` raw frames, **resizes** them to
`TARGET_H×TARGET_W` (the toy clips are `832×480`, so they are upscaled
trilinearly to `1280×704`), encodes them with the Wan2.2 VAE
(4× temporal, 16× spatial, 48 channels) into a `(F_lat=20, 48, 44, 80)`
fp16 latent, and stores per-record `latents_*`, `prompts_*`, `intrinsics_*`
(normalized `[fx/W, fy/H, cx/W, cy/H]`), `poses_*` (`[tx,ty,tz,qx,qy,qz,qw]`,
F_lat × 7). `MAX_FRAMES` must be `4*k+1` and `TARGET_H/TARGET_W` must be
divisible by 16, otherwise the script exits early. `clips.json` already stores
absolute `video_path`s, so `VIDEO_DIR` can be left unset.

> **Auto-resume.** The build is resumable by default — just re-run the exact
> same command and it continues where it stopped. Each clip's `video_path` is
> stored in the merged `data/` LMDB, which is the authoritative record of what
> is already done; on a re-run every rank reads it, **skips** the clips already
> encoded, and only **appends** the new ones (you'll see
> `[resume] N clips already processed ... skipped` and a final
> `Done! <total> samples (<new> new)`). The per-rank `.rank_*` dirs are only a
> transient parallel-write buffer (needed because LMDB allows a single writer
> per env) and are deleted after the merge; a run killed mid-encoding still
> resumes because that leftover shard is folded in on the next run. Controls:
> `NO_RESUME=1` wipes `data/` + shards and reprocesses from scratch;
> `KEEP_SHARDS=1` keeps the `.rank_*` dirs after merging (debugging only).
>
> ⚠️ If you have a `data/` produced by an **older** version of this script (it
> did not store `video_path`), resume can't tell which clips were done — run
> once with `NO_RESUME=1` to rebuild it, after which all later resumes work.

> **Lighter (native-resolution) option.** To avoid upscaling and roughly
> halve the token count / memory, build at the clips' native `832×480` with
> `TARGET_H=480 TARGET_W=832` (both divisible by 16 → latent grid `30×52`,
> still even so PRoPE token grouping stays exact). If you do this, set
> `data.image_or_video_shape: [1, 20, 48, 30, 52]` in
> `configs/train_bidir_camera.yaml` and `configs/infer_bidir_camera.yaml`
> accordingly; the dynamic seq-len in `CameraWanDiffusionWrapper` handles the
> rest. The default below keeps `1280×704` to match the shipped configs.

`model_kwargs.use_camera` are set to `false`.


Why this is exactly "camera removed":

- With `use_camera: false`, `add_prope_parameters` is **never** called, so the
  model has **no `prope_o` parameter** and is the original Wan2.2-TI2V-5B.
- The PRoPE residual in `wan_5b/modules/model.py` is gated by
  `if prope_meta is not None and getattr(self, "prope_o", None) is not None:`,
  so even though the trainer still reads `viewmats`/`Ks` from the LMDB and
  passes them down, that whole path is skipped — the camera info is silently
  ignored. **No data rebuild is needed**; the existing LMDB (latents + prompts)
  is reused as-is.
- `generator.model.requires_grad_(True)` trains the **entire 5B backbone**, and
  the loss is the per-frame shared-timestep flow-matching loss — i.e. standard
  full-parameter SFT. Pretrained `wan_models/Wan2.2-TI2V-5B/` weights are the
  starting point (add `generator_ckpt: <path>` to the config to init from a
  different checkpoint).

Checkpoints land at `logs/train_bidir_sft/checkpoint_model_XXXXXX/model.pt`;
re-running the same command auto-resumes from the latest (pass
`--no-auto-resume` to start fresh).

#### (3c) T2V ↔ I2V switch (single-line config flip)

Both `train_bidir_camera.yaml` (camera + PRoPE) and `train_bidir_sft.yaml`
(plain) accept a single `algorithm.i2v` switch that flips between T2V and I2V
SFT without any code changes. The mechanics are exactly equivalent to how
**Causal I2V AR** (`configs/train_i2v_ar.yaml`, see `model/diffusion.py` and
`utils/i2v_conditioning.py`) treats its first conditioning latent — only
ported to the bidirectional flow-matching loop.

| Stage | T2V config | I2V config |
| --- | --- | --- |
| Camera (PRoPE) SFT | `configs/train_bidir_camera.yaml` (`i2v: false`) | `configs/train_bidir_camera_i2v.yaml` (`i2v: true`) |
| Plain SFT | `configs/train_bidir_sft.yaml`     (`i2v: false`) | `configs/train_bidir_sft_i2v.yaml`     (`i2v: true`) |
| Inference | `configs/infer_bidir_camera.yaml`  (`i2v: false`) | `configs/infer_bidir_camera_i2v.yaml`  (`i2v: true`) |

When `algorithm.i2v: true`:

- **Training** (`trainer/camera_bidirectional_diffusion.py` →
  `model/camera_bidirectional_diffusion.py`):
  - `clean_latent[:, :1]` is taken as the I2V context latent (the first
    latent of the LMDB record is exactly the VAE-encoded source frame, so
    the existing `CameraLatentLMDBDataset` is reused **as-is**, no data
    rebuild needed).
  - Under sequence parallelism, the context is only applied on the SP rank
    that actually owns global frame 0 (`sp_rank == 0`); other SP ranks
    treat their local frame 0 as a normal training frame.
  - That frame's `timestep` is forced to 0 in the schedule, and
    `noisy_latents[:, :1]` is overwritten with the clean image latent
    before the backbone forward (so the model never sees the conditioning
    frame noised).
  - The flow-matching loss is computed with a per-element mask that zeros
    out the conditioning frame, exactly mirroring `_zero_i2v_context_timestep`
    + `_overwrite_i2v_context` from the Causal I2V path.

- **Inference** (`scripts/inference/inference_bidir_camera.py`):
  - Reads one extra path file `inference.image_path` (one source-image
    path per line, aligned 1-to-1 with `prompt_path` / `trajectory_path`).
  - VAE-encodes each source image with `WanVAEWrapper.encode_to_latent`
    into a single-frame latent and pins it into `x[:, :1]`.
  - Sets `timestep[:, 0] = 0` per step so the backbone treats that frame
    as clean, and **re-pins** `x[:, :1] = image_latent` after every
    `scheduler.step` so the conditioning frame stays bit-identical to the
    encoded source image throughout denoising.

Causal-vs-bidirectional I2V differences (so the contract is unambiguous):

- **Causal I2V** (`configs/train_i2v_ar.yaml`): the first chunk has
  `independent_first_frame=True` (1 latent), the dataset returns the
  source image (`return_image=True`), `noise_augmentation` is also pinned
  via `_overwrite_i2v_context` on `clean_x` for teacher forcing.
- **Bidirectional I2V** (this stage): the model is non-causal and uses a
  **single shared timestep** across all frames; there is no `clean_x`
  teacher-forcing path and no per-block scheduling, so the only places
  the I2V contract has to be enforced are (1) timestep zero on frame 0,
  (2) `noisy_latents[:, :1]` overwrite, and (3) loss mask. That is what
  the new code does — nothing more.

##### Memory footprint at 1280×704 (full bidirectional attention)

The image-shape `[1, 20, 48, 44, 80]` (latent `44×80` ≈ raw `704×1280`) is
patchified by `patch_size=(1,2,2)` into a token grid of
`20 × 22 × 40 = 17,600` tokens. Because the model is **non-causal
(bidirectional)**, every layer does dense self-attention over all 17.6k tokens.
Attention itself uses **flash-attention** (the `L×L` score matrix is *not*
materialized — memory there is ~linear in `L`), so the real cost is the
**layer activations**: the current trainer does **not** enable gradient
checkpointing, so the activations of all 32 layers (`dim=2048`, `ffn_dim=8192`)
are kept for the backward pass.

Rough per-GPU estimates (bf16 compute, FSDP `hybrid_full` sharding, AdamW fp32
states + EMA, `batch_size=1` per GPU — order-of-magnitude only; actual numbers
vary with PyTorch / FSDP / flash-attn versions):

| Setup | Resolution | Tokens | Activations (B=1) | Sharded weights+grad+optim+EMA | Per-GPU peak |
| --- | --- | ---: | ---: | ---: | ---: |
| 8× GPU, no grad-ckpt | 1280×704 | 17,600 | ~40 GB | ~12 GB / GPU | **~55–70 GB** |
| 8× GPU, grad-ckpt on | 1280×704 | 17,600 | ~8–12 GB | ~12 GB / GPU | **~30–40 GB** |
| 8× GPU, no grad-ckpt | 832×480 | 7,800 | ~18 GB | ~12 GB / GPU | **~32–42 GB** |
| 1× GPU | 1280×704 | 17,600 | ~40 GB | ~90 GB (unsharded) | **>80 GB → OOM** |

Practical guidance:

- **1280×704 needs 80 GB-class GPUs (A100/H100 80 GB), ~8 of them.** Per-GPU
  peak is roughly **55–70 GB** with `batch_size=1`; it fits but leaves little
  headroom. FSDP shards the AdamW fp32 states (the dominant static cost: 5B ×
  (master 4 B + m 4 B + v 4 B) ≈ 60 GB) across ranks, so **more GPUs = lower
  per-GPU memory** for the optimizer/weights — but the ~40 GB of activations is
  per-GPU and does *not* shrink with more ranks (only with grad-ckpt or a
  smaller sequence).
- A **single GPU cannot hold the unsharded model** (weights 10 + grads 10 +
  AdamW 60 + EMA 10 ≈ 90 GB before activations), so the 1-GPU command is a
  pipeline smoke test only — expect OOM at 1280×704 unless you also drop to
  832×480 and enable gradient checkpointing.
- **To cut memory:** (1) use the native-`832×480` build from step (2b) and set
  `data.image_or_video_shape: [1, 20, 48, 30, 52]` (≈0.44× the tokens, ~half
  the activations); (2) enable gradient checkpointing on the backbone
  (`generator.model._set_gradient_checkpointing(..., True)` / set
  `gradient_checkpointing=True` — not yet wired through the config, a one-line
  change in the wrapper); (3) add more GPUs to further shard the optimizer
  state.

#### (4) Inference

Author one prompt per line in `prompts/prompts.txt` and one matching
trajectory string per line in `prompts/trajectories.txt` (samples are
provided). Then:

```bash
CKPT=./checkpoints/LongLive-2.0-5B-Camera/model.pt \
OUT=videos/camera_bidir \
    bash scripts/inference/run_infer_bidirectional_camera.sh

# or directly:
CUDA_VISIBLE_DEVICES=0 python scripts/inference/inference_bidir_camera.py \
    --config_path  configs/infer_bidir_camera.yaml \
    --generator_ckpt logs/train_bidir_camera/checkpoint_model_000200/model.pt \
    --output_dir videos/camera_bidir

```

Outputs are `.mp4` clips at 24 fps (`F_lat=20` latent frames decode to
77 raw RGB frames at `1280×704`), one per `(prompt, trajectory)` pair, with
a sibling `.txt` recording the prompt and trajectory used.

#### Notes on the implementation

- PRoPE is added as a **zero-init residual** path (`prope_o = nn.Linear(dim, dim)`
  with weight & bias zero), so at step 0 the model is bit-exact equal to the
  pretrained Wan2.2-TI2V-5B regardless of `viewmats`. Setting
  `algorithm.use_camera = false` (or omitting `viewmats`) bypasses the path
  entirely.
- All frames in a batch share a single timestep (`uniform_timestep = True`).
- Training and inference use the same `utils.camera_dataset.build_viewmats_and_Ks`
  conversion, so PRoPE receives identical first-frame-normalized view matrices
  and normalized intrinsics in both paths.
- The new trainer (`trainer/camera_bidirectional_diffusion.py`) only relies
  on LongLive's own `utils/distributed.fsdp_wrap`, `EMA_FSDP`, and
  `fsdp_state_dict`, so it composes with the rest of the repo with no
  changes to existing AR / DMD trainers.

## DreamX-World 风格 Camera 模块集成

LongLive 在保留原有 PRoPE residual (`prope_o`) 路径的同时，并行新增了一条 **DreamX-World**
风格的 camera 路径，可以直接加载官方 `DreamX-World-5B-Cam` 权重，并支持 DreamX 的 action DSL
轨迹解析。两条路径通过 yaml 开关 `algorithm.dreamx_camera` 互斥切换。

### 结构差异

| | LongLive 原 PRoPE residual | DreamX-World 路径（本次新增） |
|---|---|---|
| 模块位置 | `WanSelfAttention.prope_o` (内嵌于 RoPE self-attn) | `WanAttentionBlock.cam_self_attn` (与 self-attn 并联) |
| 模块结构 | 单个 `Linear(dim,dim)`，zero-init | 完整 `PropeSelfAttention`：q_proj/k_proj/v_proj/out_proj + RMSNorm，out_proj zero-init |
| 计算式 | `out = self.o(x_rope) + self.prope_o(x_prope)` | `y = self_attn(x_rope) + cam_self_attn(x, cam_emb)` |
| ckpt 命名 | `blocks.{i}.self_attn.prope_o.*` | `blocks.{i}.cam_self_attn.{q,k,v,out}_proj.*` + `norm_q/norm_k.weight` |

PRoPE 数学（`prope_qkv`、`_invert_SE3`、`_lift_K`、`_invert_K`）两条路径完全共享，落地在
`wan_5b/modules/prope.py`。

### 新增/修改的文件

| 文件 | 作用 |
|---|---|
| `wan_5b/modules/dreamx_camera.py` | `PropeSelfAttention` 类 + `add_dreamx_cam_self_attn(model)` helper，按 `blocks.{i}.cam_self_attn.*` 命名挂载 |
| `wan_5b/modules/model.py` | `WanAttentionBlock.forward`：在 `y = self.self_attn(temp_x, ...)` 之后并联 `y += self.cam_self_attn(temp_x, cam_emb, seq_lens)` |
| `utils/dreamx_camera_wrapper.py` | `DreamXCameraWanDiffusionWrapper`：`CameraWanDiffusionWrapper` 子类；构造时调 `add_dreamx_cam_self_attn`；提供 `load_dreamx_checkpoint(dir)` 与 `freeze_backbone()` |
| `utils/dreamx_trajectory.py` | DreamX action DSL 解析：`action_to_viewmats_Ks(action_seq, action_speed_list, duration, target_length, h, w)` → `(viewmats, Ks)`；含 `interpolate_camera_poses`（SLERP+LERP，对齐 VAE 时序步长 4） |
| `model/camera_bidirectional_diffusion.py` | 训练时根据 `algorithm.dreamx_camera` 切换 wrapper；`freeze_backbone=true` 时只训 `cam_self_attn` |
| `scripts/inference/inference_bidir_camera.py` | 推理脚本：增加 `dreamx_camera` 分支、`trajectory_format=dreamx_action_dsl`，沿用 worldplaygen-style 的 `prompt_path/trajectory_path/image_path` 三件套；DSL 行格式 `"w-4,a-4,..."` 由 `parse_trajectory_string` 解析 |
| `configs/infer_dreamx_camera_i2v.yaml` | 推理配置示例（与 `inference_dreamx_5b_yyx.sh` 等价） |
| `configs/train_dreamx_camera_i2v.yaml` | 训练配置示例（DreamX 风格：冻 backbone，仅训 cam_self_attn） |

### 推理（与 DreamX 原版等价）

输入沿用 worldplaygen-style 的三件套（line-aligned 文本文件，与现有
`configs/infer_bidir_camera_i2v.yaml` 完全同形），只是 `trajectory_path`
中的每行用 DreamX action DSL 表达：``"<token>-<speed>,..."``，token 可
以是 `w/a/s/d/i/k/j/l` 单字母或组合（如 ``"wl"``）。

```text
# prompts/dreamx_prompts.txt          (一行一个 caption)
# prompts/dreamx_trajectories.txt     (一行一个轨迹串，如 "w-4,a-4,w-4,d-4")
# prompts/dreamx_images.txt           (一行一个 source image 路径，I2V 用)
```

`eval_demo.json` 的 7 条记录已自动落到 `prompts/dreamx_*.txt`；其他
DreamX JSON 可用一行 Python 转写：

```python
import json
recs = json.load(open('your.json'))
open('prompts/dreamx_trajectories.txt','w').write('\n'.join(
    ','.join(f'{tok}-{sp:g}' for tok,sp in zip(r['action_seq'], r['action_speed_list']))
    for r in recs) + '\n')
```

启动命令：

```bash
CUDA_VISIBLE_DEVICES=0 python scripts/inference/inference_bidir_camera.py \
    --config_path  configs/infer_dreamx_camera_i2v.yaml \
    --output_dir   videos/dreamx_camera_i2v
```

加载权重的优先级：
1. 若 `--generator_ckpt` 给了 LongLive 自训的 ckpt（应同时含 backbone + `cam_self_attn`），用它。
2. 否则用 `model_kwargs.dreamx_ckpt`（默认指向 `DreamX-World-5B-Cam` 目录）。
3. 都没有时，backbone 取 `wan_models/Wan2.2-TI2V-5B`、`cam_self_attn` 全部 zero-init。

### 训练

DreamX 训练 recipe：冻整个 Wan2.2-TI2V-5B backbone，只让 `cam_self_attn` 参与梯度更新。
LongLive 这边在 `_initialize_models` 里通过 `freeze_backbone_for_train=True` 自动完成冻结，
trainer 端不需要任何改动（trainer 只与 wrapper 交互）。

```bash
torchrun --nproc_per_node=8 train.py \
    --config_path configs/train_dreamx_camera_i2v.yaml
```

数据集复用 `CameraLatentLMDBDataset`：LMDB 已经存了 `latents / prompts / intrinsics / poses`，
`build_viewmats_and_Ks` 把它们转成 `(viewmats, Ks)` 后由 wrapper 注入 `prope_meta`，
对 DreamX 路径无差别可用。

### Ablation

* `NO_PROPE=1 python scripts/inference/inference_bidir_camera.py ...`：把每个 block 的
  `cam_self_attn.out_proj` 权重清零，等价于关闭 camera 控制，验证视频质量是否退化到 base
  Wan2.2-TI2V-5B 输出。
* 把 `algorithm.dreamx_camera` 翻回 `false`，立刻退回 LongLive 原 PRoPE residual 路径，
  其余推理逻辑（I2V pin first frame、scalar t scheduler step 等）完全一致。
